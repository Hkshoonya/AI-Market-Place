  <name>Task 1: Write model detail page E2E test</name>
  <files>e2e/model-detail.spec.ts</files>
  <action>
    Create `e2e/model-detail.spec.ts` with test.describe('Model detail page'):

    **Architecture note:** The model detail page is a Server Component. Supabase queries happen server-side and CANNOT be intercepted by Playwright. The test strategy is:
    - If the dev server has valid Supabase env vars: pages render with real data, tests verify real content
    - If not: pages may show "not found" or empty state, tests should be resilient

    For maximum reliability, use a known model slug that likely exists in the database. If the page 404s, skip gracefully. Alternatively, test the page structure that renders regardless of data.

    **Recommended approach:** Navigate to `/models` (the discover page which lists models), find the first model link, click it, then verify the detail page structure. This avoids hardcoding a slug. If no models exist, skip the test.

    Tests:

    1. **Test: 'model detail page displays model name and stats'**
       - Navigate to `/discover` or use a common slug like 'gpt-4o' (fallback: skip if 404)
       - Wait for page to load (wait for heading element)
       - If page has a model name heading, assert it's visible
       - Look for stats row content (quality score, market cap display)
       - Use `test.skip()` condition if page shows 404/not-found

    2. **Test: 'tab navigation switches content panels'**
       - Navigate to model page (same approach as test 1)
       - Default tab is "benchmarks" -- verify the benchmarks content area is visible
       - Click "Pricing" tab: `page.getByRole('tab', { name: 'Pricing' }).click()`
       - Verify pricing content becomes visible (or the tab has aria-selected="true")
       - Click "Details" tab: `page.getByRole('tab', { name: 'Details' }).click()`
       - Verify details content becomes visible
       - Click "Deploy" tab and verify
       - These are Radix UI Tabs, so `aria-selected` attribute changes on click

    3. **Test: 'clicking model name link from another page navigates to detail'**
       - Navigate to `/leaderboards`
       - Find first model link in the page (Link elements pointing to /models/*)
       - Click it
       - Verify URL matches /models/<slug> pattern
       - Verify page renders model content (heading visible)

    Use `page.getByRole('tab', { name: ... })` for tab selectors.
    Use `expect(tabTrigger).toHaveAttribute('aria-selected', 'true')` to verify active tab.
    Use `test.skip()` if model data is unavailable to avoid false failures in CI without env vars.
  </action>

  <name>Task 2: Write leaderboard E2E test</name>
  <files>e2e/leaderboard.spec.ts</files>
  <action>
    Create `e2e/leaderboard.spec.ts` with test.describe('Leaderboard'):

    **Architecture note:** Same as model detail -- the leaderboard page is server-rendered. The `LeaderboardExplorer` client component receives `explorerModels` as props from the RSC. Lens switching, sorting, and filtering are purely client-side (useState in LeaderboardExplorer, @tanstack/react-table). These client-side interactions DO NOT make new API calls -- they re-sort/re-filter the props array in memory.

    Tests:

    1. **Test: 'leaderboard page loads with Explorer tab active'**
       - Navigate to `/leaderboards`
       - The default tab is "explorer" (Radix Tabs defaultValue="explorer")
       - Verify Explorer tab trigger has `aria-selected="true"`: `await expect(page.getByRole('tab', { name: 'Explorer' })).toHaveAttribute('aria-selected', 'true')`
       - Verify some content is visible in the explorer panel (look for table headers or model rows)
       - If no models rendered (empty DB), test should still pass by checking the tab itself is active

    2. **Test: 'switching lens changes active lens indicator'**
       - Navigate to `/leaderboards`
       - The explorer tab should be active by default
       - The lens buttons are rendered by LeaderboardControls -- they're `<button>` elements with text "Capability", "Usage", "Expert", "Balanced"
       - Default lens is "capability" (useState initial value)
       - Click "Usage" lens button: `page.getByRole('button', { name: 'Usage' }).click()` (these are plain buttons, not Radix tabs)
       - Verify the "Usage" button has the active styling (bg-white class). Since we can't rely on class names, check that the button visually changed by verifying the button is still present and was clicked. Alternatively use a locator chain.
       - Click "Expert" and verify
       - Click "Balanced" and verify
       - Note: The lens buttons change the sort column in the table. If models exist, verify the table re-renders (row order may change). If no models, just verify buttons are clickable.

    3. **Test: 'main tab navigation works across leaderboard views'**
       - Navigate to `/leaderboards`
       - Click "Top 20" tab (value="overall"): `page.getByRole('tab', { name: 'Top 20' }).click()`
       - Verify tab is selected: `aria-selected="true"`
       - Click "Speed" tab: `page.getByRole('tab', { name: 'Speed' }).click()`
       - Verify tab is selected
       - Click "Best Value" tab: `page.getByRole('tab', { name: 'Best Value' }).click()`
       - Verify tab is selected
       - Click back to "Explorer": `page.getByRole('tab', { name: 'Explorer' }).click()`
       - Verify Explorer is selected again

    4. **Test: 'category filter badges are visible'**
       - Navigate to `/leaderboards`
       - The page renders category badges ("All Models" plus CATEGORIES entries)
       - Verify "All Models" badge is visible: `await expect(page.getByText('All Models')).toBeVisible()`
       - Verify at least one category badge is visible (e.g., look for category short labels)

    Use `page.getByRole('tab', { name: ... })` for the main Radix tabs.
    Use `page.getByRole('button', { name: ... })` for lens switching buttons (they are plain HTML buttons inside LeaderboardControls).
    Handle empty data gracefully -- the test should pass even if no models are in the database.
