  <action>
    Create `e2e/model-detail.spec.ts` with test.describe('Model detail page'):

    **Architecture note:** The model detail page is a Server Component. Supabase queries happen server-side and CANNOT be intercepted by Playwright. The dev server uses dummy Supabase env vars, so server-side queries will fail. Tests verify page shell structure and client-side interactive behavior only.

    **Strategy:** Since server-side data fetching will fail with dummy env vars, the page may render an error/not-found state. The approach is:
    - Use `test.skip()` if the page returns a 404 or completely fails to render
    - If the page renders ANY shell (even with empty data sections), test the interactive elements
    - Intercept client-side SWR API calls (deploy-tab, model-overview) with fixtures via mockApiRoute
    - A fallback approach: navigate to `/leaderboards`, find a model link, try clicking it -- but this also depends on server data

    **Recommended approach for offline tests:** Navigate to a known slug (e.g., `/models/gpt-4o`). If the page renders at all (even partially), verify:

    Tests:

    1. **Test: 'model detail page renders page shell with tabs'**
       - Set up route interception for client-side SWR calls BEFORE navigation:
         * `mockApiRoute(page, '**/api/models/*/deployments', { deployments: [], platforms: [] })`
         * `mockApiRoute(page, '**/api/models/*/description', { description: 'Test description', generated_at: '2024-01-01' })`
       - Navigate to `/models/gpt-4o` (or any slug)
       - If page responds (not a hard crash), check for any heading element on the page
       - If heading with model name is visible, assert it contains text (model name)
       - If page shows score metrics (quality_score, market_cap), assert at least one score element is visible. Look for text patterns like "Quality Score", "Market Cap", or numeric score displays.
       - Use `test.skip()` condition if the page returns a completely blank/error page with no interactive elements

    2. **Test: 'tab navigation switches content panels'**
       - Navigate to model page (same approach as test 1, with same SWR intercepts)
       - The tabs are Radix UI Tabs and render regardless of data content (they're part of the page shell)
       - Default tab is "benchmarks" -- verify the benchmarks tab has `aria-selected="true"`
       - Click "Pricing" tab: `page.getByRole('tab', { name: 'Pricing' }).click()`
       - Verify Pricing tab has `aria-selected="true"` and Benchmarks no longer does
       - Click "Details" tab: `page.getByRole('tab', { name: 'Details' }).click()`
       - Verify Details tab is now selected
       - Click "Deploy" tab: `page.getByRole('tab', { name: 'Deploy' }).click()`
       - Verify Deploy tab is selected
       - These are client-side Radix tab switches -- they work even if data sections are empty

    3. **Test: 'clicking model link from leaderboard navigates to detail'**
       - Navigate to `/leaderboards`
       - Look for any link matching `a[href*="/models/"]`
       - If links found (count > 0): click first link, verify URL matches /models/* pattern, verify page renders
       - If no links found (empty leaderboard due to no server data): `test.skip('No model links available offline')`
       - This tests navigation flow, not data content

    Use `page.getByRole('tab', { name: ... })` for tab selectors.
    Use `expect(tabTrigger).toHaveAttribute('aria-selected', 'true')` to verify active tab.
    Use `test.skip()` if page structure is unavailable to avoid false failures.
