  <action>
1. Run `npm install swr` to add SWR 2.x as a dependency.

2. Create `src/lib/swr/fetcher.ts`:
   - Export an async function `jsonFetcher<T>(url: string): Promise<T>` that calls `fetch(url)`, checks `res.ok`, throws an Error with status attached on failure, and returns `res.json()` on success.
   - Attach the HTTP status to the error object as a `status` property so consumers can differentiate 401 vs 500.

3. Create `src/lib/swr/config.ts`:
   - Export `SWR_TIERS` as a const object with three tiers:
     - `FAST`: `{ refreshInterval: 30_000, dedupingInterval: 10_000 }` -- for ticker, top movers, auction data
     - `MEDIUM`: `{ refreshInterval: 60_000, dedupingInterval: 30_000 }` -- for notifications, wallet, order status
     - `SLOW`: `{ refreshInterval: 0, dedupingInterval: 300_000 }` -- for model metadata, descriptions, deployments
   - Type the tiers with `as const` for narrowing.

4. Create `src/lib/swr/config.test.ts` (Wave 0 requirement from VALIDATION.md):
   - Import `SWR_TIERS` from './config'.
   - Test that all three tiers (FAST, MEDIUM, SLOW) exist and have `refreshInterval` and `dedupingInterval` as numbers.
   - Test that FAST.refreshInterval < MEDIUM.refreshInterval (or MEDIUM > 0 while SLOW is 0).
   - Test that SLOW.refreshInterval is 0 (no polling for static data).
   - This is a lightweight sanity test ensuring the tier constants are well-typed and logically ordered.

5. Create `src/lib/swr/test-utils.ts`:
   - Export a `renderWithSWR(ui: React.ReactElement)` function that wraps the component in `<SWRConfig value={{ provider: () => new Map(), dedupingInterval: 0 }}>` and calls Testing Library's `render()`.
   - This ensures each test gets a fresh SWR cache (official SWR testing pattern from https://swr.vercel.app/docs/advanced/cache).
   - Import `SWRConfig` from 'swr' and `render` from '@testing-library/react'.
