<objective>
Fix Zod schema type mismatches causing model detail 404s, admin analytics crashes, and leaderboard empty states

Purpose: Postgres returns numeric and bigint column values as strings via PostgREST/Supabase JS. Zod z.number() rejects these strings, causing parseQueryResult/parseQueryResultSingle to silently return []/null. Additionally, is_open_weights has no NOT NULL constraint in DB but schemas define it as z.boolean() (non-nullable). These two systematic mismatches cause UAT test failures 2, 3, 5, and 7.

Output: All schema files updated with z.coerce.number() for numeric fields and z.boolean().nullable() for is_open_weights, plus admin analytics error handling.
</objective>

From src/lib/schemas/parse.ts:
```typescript
export function parseQueryResult<T>(response: SupabaseResponse, schema: z.ZodType<T>, schemaName: string): T[];
export function parseQueryResultSingle<T>(response: SupabaseResponse, schema: z.ZodType<T>, schemaName: string): T | null;
```

ROOT CAUSE: PostgREST returns Postgres numeric/bigint as strings (e.g., "92.3" not 92.3).
z.number() rejects "92.3" -> safeParse fails -> returns []/null.
z.coerce.number() accepts "92.3" and converts to 92.3.

SECONDARY CAUSE: is_open_weights column has DEFAULT false but NO NOT NULL constraint.
z.boolean() rejects null -> safeParse fails -> returns []/null.
z.boolean().nullable() accepts null.
